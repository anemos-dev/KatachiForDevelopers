# Katachi for Developers 仕様書（MVP + β）

最終更新: 2026-04-11

## 1. 目的

開発中に生まれる断片的な発想を、**数秒で保存し、あとで実装候補として再利用できる形で管理する**。

Katachiは「曖昧な着想を、次に進められる形へ変える」ためのプロダクトファミリーとする。初回リリースは **Katachi for Developers** として、開発者の思考フローに最適化したカード管理アプリにする。

将来的には同じKatachiブランドで **Katachi for Designers** などへ展開できるよう、コア体験は「保存」「構造化」「再発見」「次アクション化」に寄せる。

## 2. プロダクト方針

- 入力速度と再発見性を両立する
- 1メモを「1アイデアカード」として扱う
- 本文はプレーンテキストで運用する
- 入力項目を分割し、空白の巨大メモ欄を作らない
- ステータス管理で「保存して終わり」を防ぐ
- ホーム画面表示名は `Katachi`、App Store上の名称は `Katachi for Developers` とする

## 3. スコープ

### 3.1 MVPに含む

- アイデア作成
- 一覧カード表示
- 詳細編集
- 検索
- フィルタ（種類 / ステータス / お気に入り）
- 並び替え（更新日 / 作成日 / 優先度）
- タグ管理（カンマ区切り）
- お気に入り

### 3.2 MVPで含まない

- AI要約 / AI分類
- iCloud同期
- マルチデバイス同期
- Markdownレンダリング
- 添付ファイル
- コラボ編集

## 4. データ仕様

## 4.1 エンティティ: Idea

- `id: UUID`（一意）
- `title: String`（表示タイトル）
- `kindRaw: String`（種類）
- `statusRaw: String`（ステータス）
- `concept: String`（何を思いついたか）
- `rationale: String`（なぜ良さそうか）
- `approach: String`（どう作るか / どう試すか）
- `nextAction: String`（次にやること）
- `projectName: String`（任意）
- `tags: [String]`（タグ配列）
- `priority: Int`（1〜5）
- `isFavorite: Bool`
- `createdAt: Date`
- `updatedAt: Date`

## 4.2 種類（IdeaKind）

- `feature`（機能案）
- `uiux`（UI改善）
- `tech`（技術調査）
- `bug`（バグ仮説）
- `article`（記事ネタ）
- `note`（雑メモ）

## 4.3 ステータス（IdeaStatus）

- `inbox`
- `refining`
- `ready`
- `done`
- `hold`

## 4.4 保存ルール

- 保存条件: `title` か `concept` のどちらかが非空
- `title` が空の場合、`concept` の先頭行を仮タイトルとして自動生成
- `priority` のデフォルトは `3`
- `status` のデフォルトは `inbox`
- `kind` のデフォルトは `feature`
- タグはトリムして重複排除（大文字小文字は区別しない）

## 5. 画面仕様

## 5.1 一覧画面（Home）

表示要素:

- 検索バー
- フィルタボタン
- 並び替えメニュー
- カード一覧
- 新規作成ボタン（+）

空状態:

- カード0件時はKatachiに最初のカードを保存する導線を表示

## 5.2 一覧カード

表示順:

1. タイトル（最大2行）
2. `concept` プレビュー（最大2行）
3. ステータスバッジ + 種類バッジ
4. タグ（最大3件、超過時 `+N`）
5. `nextAction`（存在時のみ1行）
6. 更新日時（相対表示）

補助表示:

- お気に入り（右上アイコン）

操作:

- タップで詳細へ遷移
- スワイプで削除
- スワイプでお気に入り切替

## 5.3 新規作成 / 編集画面

初期表示項目（基本）:

- タイトル
- 種類
- 何を思いついたか（concept）
- なぜ良さそうか（rationale）

詳細項目（折りたたみ）:

- どう作るか / どう試すか（approach）
- 次にやること（nextAction）
- タグ（カンマ区切り）
- ステータス
- 優先度（1〜5）
- 関連プロジェクト名（projectName）
- お気に入り

ボタン:

- `Cancel`
- `Save`（保存条件を満たす時のみ活性）

## 5.4 フィルタ画面

条件:

- 種類（単一選択 / すべて）
- ステータス（単一選択 / すべて）
- お気に入りのみ

操作:

- `条件をリセット`
- `適用`

## 6. 検索仕様

検索対象:

- タイトル
- concept
- rationale
- approach
- nextAction
- projectName
- tags

仕様:

- 大文字小文字を区別しない
- 部分一致

## 7. 並び替え仕様

- `更新日（新しい順）`
- `作成日（新しい順）`
- `優先度（高い順 -> 更新日）`

## 8. 表示ルール

- 日付は相対表示優先（例: 今日 / 2日前）
- バッジ色はステータスを最優先で意味づけ
- 色に依存しすぎずラベル文字も表示

## 9. 非機能要件（MVP）

- 2000件程度のカードで一覧スクロールが実用速度
- 主要操作（作成/保存/検索）でアプリがクラッシュしない
- オフライン前提で全機能利用可
- 入力中の画面遷移でデータ破損しない

## 10. β受け入れ基準

以下を満たしたらβ配布可能と判断する。

- 新規作成、編集、削除が正常動作
- 検索、フィルタ、並び替えが意図通り反映
- カード表示が仕様通り（省略、タグ上限、次アクション条件表示）
- ステータス変更とお気に入り切替が保持される
- 空データ時と大量データ時で表示破綻がない
- 最低1回の端末再起動後にデータが保持される

## 11. βテスト項目

### 11.1 体験品質

- 思いつき保存にかかる時間（目標: 20秒以内）
- 一覧での再発見しやすさ（主観評価）
- 項目分割により記入しやすくなっているか

### 11.2 機能品質

- タグ正規化（余分な空白や重複の処理）
- 仮タイトル自動生成
- 絞り込みと検索の同時利用
- ソート切替時の一覧再描画

### 11.3 運用品質

- TestFlightでの初回起動成功率
- クラッシュ有無
- 致命的UI崩れの有無（小画面端末）

## 12. β期間の収集フィードバック

- 「保存は速いか」
- 「一覧だけで思い出せるか」
- 「入力項目の多さは負担か」
- 「ステータス定義は自然か」
- 「次アクション欄が役立つか」

この5点を中心に次バージョンの改善優先度を決める。
